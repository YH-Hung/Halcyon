# Halcyon v1.2 — Coroutines & Async Streaming — Design Spec

**Date:** 2026-07-16 · **Status:** approved (design) · **Baseline:** v1.1.0
(2026-07-13)

## 1. Overview

v1.2 delivers the marquee item deferred since the original design: the C++20
coroutine layer the executor seam was built "coroutine-ready" for, including
the async streaming follow-up (`queryStreamingAsync`) recorded in the v1.1
spec §10. One operational ride-along completes the backlog's last small item.

1. **C++20 coroutine layer** — `halcyon::coro::Task<T>` plus awaitable forms
   of `execute`, `query`, `executeBatch`, `transaction`, and streaming
   (awaitable row fetch and LOB chunk reads).
2. **Pool health stats** — a `PoolStats` snapshot accessor on
   `ConnectionPool` and `Database`.

**Approach (user-approved):** executor-offload awaitables. Each awaited
operation submits the existing blocking call to the existing `Executor`
thread pool; the awaiting coroutine suspends and is resumed on the completing
worker thread. The layer is a **C++20 adapter header plus a small additive
C++17 support surface** (§A.1): one generic async-offload hook on `Database`
and worker-thread detection on `Executor`, both in the existing C++17
headers. The library still builds as C++17; review of the initial draft
established that a purely external header cannot work — `Database::exec_` is
private, the future-based lambdas use private machinery (`instrument`,
retry-policy helpers, tracer capture), and batch/transaction/streaming have
no future APIs to wrap. Rejected alternatives: a
per-connection async actor/strand subsystem (large new machinery with no real
concurrency gain over a blocking CLI) and caller-supplied resume schedulers /
sender-receiver shapes (bigger abstract surface; can be layered on later
without breaking changes).

### Goals

- Full async parity: everything the `std::future` API covers, plus streaming,
  awaitable from C++20 coroutines.
- Library contracts intact: dual error model, RAII/single-owner semantics,
  thread-safe pool, auto-retry/metrics/tracing/logging inherited unchanged
  (each awaitable *is* the existing sync path, offloaded).
- The library's C++17 requirement is unchanged; existing headers gain only a
  small additive support surface, and the coroutine layer itself is a
  separate opt-in header for C++20 consumers.
- Zero new dependencies. SemVer **minor**: purely additive, version `1.2.0`.

### Non-goals (explicitly out of scope)

- Cancellation tokens / stop sources for in-flight operations.
- Caller-supplied resume schedulers; asio/libcoro/stdexec interop.
- Raising the library's own build standard to C++20.
- Generator-style `co_for` row iteration (use the `next()` loop).
- LOB support in `executeBatch`; scrollable cursors; vcpkg/Conan submission;
  Windows/AIX; versioned docs (all remain on the recorded backlog).

## 2. Decisions (locked)

| Topic | Decision |
|---|---|
| Version | `1.2.0`, SemVer minor, purely additive |
| Task type | Own dependency-free `halcyon::coro::Task<T>` (lazy, move-only, single-consumer, symmetric transfer); no third-party runtime |
| Execution model | Offload to the existing `Executor`; resume on the completing worker thread |
| Packaging | New header `halcyon/coro.hpp`, **not** in the `halcyon.hpp` umbrella; clear `#error` without C++20 coroutine support |
| C++17 support surface | Additive, in existing headers: `Database::asyncBacking()` (factory-time bundle: pool-only sync view carrying the owned-driver keep-alive, weak executor state, trace context captured at the call), `Executor::post(fn)` (fire-and-forget, **noexcept-callable contract**), `Executor::onWorkerThread()` |
| Executor teardown | `Executor` is refactored into a thin handle over a `shared_ptr`'d internal `State` (queue, mutex, cv, stop flag, worker ids); worker threads co-own the `State`. `~Executor` sets stop, joins every worker **except the current thread**, and detaches itself when it *is* a worker — so running the destructor on a worker (which user continuations make possible no matter what the coro layer does) can never self-join, and the loop's post-destructor member access is backed by the co-owned `State`, not the freed handle |
| Executor ownership | Coroutine frames hold the `AsyncBacking` bundle — never a strong reference to the `Executor` handle. Awaits lock a `weak_ptr<Executor::State>`: a strong `State` reference controls no thread lifetimes, so releasing it anywhere (including on a worker) can at most free the queue block, never join. Lock failure (backing gone) completes the await with `ErrorCode::InvalidState`. Drain-on-last-copy is preserved |
| Driver keep-alive | `AsyncBacking`'s sync view retains `owned_driver_` ordered to outlive its pool reference — the same declaration-order pattern as `QueryResult` — because `ConnectionPool` holds a raw `ICliDriver*` |
| Parameter ownership | Factories materialize ordinary parameters into owned seam `Value`s at call time (`string_view`/`const char*` are copied into owning storage then). `LobSource` **wrappers are owned by the frame** (temporaries like `lobFile(path)` are safe); only their **referents** (`lobStream`'s stream, reference captures of `lobCallback`) are borrowed and must outlive the task |
| ODR safety | No `#if`-gated members on existing classes; all coroutine API is free functions/types in the new header, namespace `halcyon::coro` |
| Error model | All ops return `Task<Result<...>>`; throwing style is the documented idiom `(co_await …).value()` — no doubled overload set |
| Argument lifetime | Public entry points are **non-coroutine factories**: they obtain the `AsyncBacking` bundle (context capture) and materialize arguments into owned storage synchronously, then return an internal coroutine — forwarding references never reach a lazy frame |
| Transaction fn | `coro::transaction` accepts a plain `Result<T>`-returning callable **or** a `Task`-returning coroutine (trait-dispatched); after a coroutine fn completes, the scope **hops back to a Halcyon worker** before commit/rollback |
| Async streaming | `StreamingQuery` wraps the v1.1 `StreamingResultSet`; awaitable `next()` and LOB `read()`; `row.lob(col)` returns `Result<coro::LobReader>` (mirrors v1.1); scalar cell reads stay synchronous |
| Sync bridge | `coro::syncWait(task)`; forbidden on executor worker threads — documented, plus a debug assert via `Executor::onWorkerThread()` |
| Pool stats | `PoolStats` value struct; `ConnectionPool::stats()` snapshot under the pool mutex; `Database::poolStats()` forwarder |
| Seam growth | **None.** Zero new `ICliDriver` methods; the layer sits entirely above the facade |
| Compiler floor (coro tests/examples) | GCC 11+, Clang 14+, AppleClang 14+; gated on the coroutine feature-test macros, skipped cleanly below |

## 3. Workstream A — Coroutine layer

### A.1 Packaging & build

- New public header `include/halcyon/coro.hpp` (implementation may split into
  `include/halcyon/coro/*.hpp` detail headers; `coro.hpp` is the entry point).
- The coroutine layer builds on a **small additive C++17 support surface** in
  the existing headers (everything else is templates/inline in `coro.hpp`;
  no new consumer-facing CMake options):
  - `Database::AsyncBacking` + `Database::asyncBacking()` — the bridge
    between factory time and await time. The accessor runs synchronously in
    the factory and returns a movable bundle of:
    1. a **pool-only sync view** — a `Database` holding the owned-driver
       keep-alive and the shared pool but a null executor. It serves the
       full synchronous API (retry, instrument, metrics, logging), so every
       coro op — including batch, transaction, and streaming, which have no
       future-based equivalents — reuses the same instrumented paths; its
       async entry points return `ErrorCode::InvalidState`. The view's
       members are declared driver-before-pool so the pool is destroyed
       first — the same keep-alive/destruction-order pattern `QueryResult`
       documents, required because `ConnectionPool` holds a raw
       `ICliDriver*`;
    2. a `std::weak_ptr<Executor::State>` (§A.2 ownership rules);
    3. the OTel trace context, captured **here and only here** — at the
       call site, exactly as `executeAsync`/`queryAsync` capture today —
       and attached on the worker via the existing `useParentContext`.
  - `Executor` handle/`State` split — the queue, mutex, cv, stop flag, and
    worker ids move into an internal `State` block co-owned
    (`shared_ptr`) by the handle and by every worker thread. `~Executor`
    sets stop, joins every worker except the current thread, and detaches
    itself when invoked *on* a worker; the worker loop touches only the
    co-owned `State` afterwards, so no freed memory is reachable. Teardown
    is thereby safe on any thread. Semantics of `submit` are unchanged.
  - `Executor::post(fn)` — fire-and-forget submission (no
    `std::future`/`packaged_task`), used by awaitables to run the offloaded
    op and then resume the suspended coroutine handle. Enqueueing operates
    on the `State` block (the handle member forwards), so the coro layer
    posts through a locked `weak_ptr<Executor::State>` without ever
    touching the handle. Because the worker
    loop invokes jobs bare (today only `packaged_task` contains
    exceptions), `post` carries a **noexcept-callable contract**: a
    throwing job would escape the thread function and terminate the
    process. Documented, `static_assert`ed where expressible; §A.2 explains
    how coro closures satisfy it.
  - `Executor::onWorkerThread()` — thread-local worker detection, needed by
    the `syncWait` debug assert and available to users writing their own
    guards.
  - `PoolStats` (§4).
- Guard at the top of the header: if `__cpp_impl_coroutine` /
  `__cpp_lib_coroutine` are absent, `#error "halcyon/coro.hpp requires C++20
  coroutines; compile this TU with -std=c++20 or newer"`.
- The umbrella `halcyon.hpp` does **not** include it; C++17 consumers see
  nothing new.
- Test/example targets that exercise the layer compile with `cxx_std_20` and
  are excluded with a configure-time notice when the toolchain lacks
  coroutine support.

### A.2 `Task<T>` machinery

- `halcyon::coro::Task<T>`: lazy (nothing runs until first `co_await`),
  move-only, single-consumer; `Task<void>` specialization.
- Completion resumes the awaiter via **symmetric transfer** — deep task
  chains must not grow the stack (unit-tested with a ~100k-deep chain).
- Exceptions thrown inside a task propagate out of `co_await` (rethrown), so
  the throwing style composes naturally.
- A `Task` that is never awaited never runs and destroys cleanly. Once
  awaited, the consumer is suspended until completion, so mid-flight
  destruction by the consumer cannot occur; no detach/fire-and-forget API.
- **Await mechanics and the self-join defense (two layers).** Every
  offloading await follows one sequence, entirely inside `await_suspend` on
  the awaiting thread (which is itself often a worker — every continuation
  after the first await runs on one):
  1. `lock()` the bundle's `weak_ptr<Executor::State>`; on failure the
     await completes immediately with `ErrorCode::InvalidState` (no
     suspension).
  2. `post()` a closure that — on the worker — attaches the bundle's trace
     context (`useParentContext` on the sync view), runs the blocking op
     against the sync view **inside a catch-all** that stores the result or
     the captured `std::exception_ptr` into the frame, and resumes the
     coroutine handle. Completion is that explicit `resume()`; no future is
     involved. `co_await` rethrows a stored exception on resumption — this
     is what satisfies `post`'s noexcept-callable contract while preserving
     "exceptions propagate out of `co_await`".
  3. Drop the `State` reference.
  Layer one: no await path ever holds a strong reference to the `Executor`
  handle — frames, temporaries, and queued closures reference only the
  `State` block, whose release controls no thread lifetimes and is
  therefore safe on any thread at any time. (A temporary strong *handle*
  reference would not be safe even held briefly: `await_suspend` running on
  a worker could lock, lose a race with another thread destroying the last
  `Database`, and become the final release — invoking `~Executor` on the
  worker.) Layer two: teardown itself is self-safe (§A.1 handle/`State`
  split) for the case the coro layer cannot prevent — a *user* coroutine
  frame legitimately holding a `Database` copy that is destroyed on a
  worker. Consequences, all documented:
  - the drain-on-last-`Database`-copy guarantee is preserved for teardown
    off-worker; teardown *on* a worker drains the queue but detaches that
    one thread instead of joining it;
  - awaiting a task *after* the last `Database` copy is gone is well-defined:
    the lock fails and the await completes with `ErrorCode::InvalidState`
    (futures never had this case because they submit eagerly; lazy tasks
    submit at await time).
- `coro::syncWait(Task<T>)` bridges to synchronous code (blocks on a
  condition variable, returns `T` or rethrows). Calling it from an executor
  worker thread can self-deadlock when the pool is saturated: documented,
  and debug builds assert via `Executor::onWorkerThread()`.

### A.3 Awaitable operations

Every public entry point is a **non-coroutine factory function**: it runs
synchronously at the call site, where it (1) obtains the `AsyncBacking`
bundle via `db.asyncBacking()` — which is where and when the OTel trace
context is captured, (2) copies `sql` and **materializes every ordinary
parameter into an owned seam `Value`** (the variant owns `std::string`, so
`string_view`/`const char*` arguments are copied into owning storage here;
a `Value`-passthrough bind is added if the bind path lacks one), and
(3) hands the bundle and the owned values to the internal coroutine it
returns. This is a hard structural rule, not a style choice — a lazy
coroutine taking `Args&&...` would bind dangling references into its frame,
would capture the trace context on whatever thread later awaits it, and a
plain decay-copy is not ownership (`decay_t<string_view>` is still a view).

The one deliberate exception concerns `LobSource` — and the borrow is of
its **referents, not the wrapper**. The wrapper object itself is moved into
the coroutine frame like any other parameter, so passing a temporary
(`coro::execute(db, sql, id, halcyon::lobFile("report.pdf"))`) is safe and
is the expected idiom; `lobFile`'s path string is owned by the wrapper, so
that call borrows nothing at all. What *is* borrowed is what a wrapper
references: `lobStream`'s `istream&`, and whatever a `lobCallback` callable
captures by reference. The contract, stated in the header docs and the
guide: those referents must stay alive until the returned `Task` completes.

Because factories hold only the bundle (§A.2), an in-flight `Task` completes
safely after its `Database` is destroyed (drain, with the sync view keeping
pool and driver alive in order), and a not-yet-awaited one degrades to
`ErrorCode::InvalidState` instead of undefined behavior.

```cpp
namespace halcyon::coro {

Task<Result<std::int64_t>>   execute(Database db, std::string sql, Args&&... args);

template <class T>
Task<Result<std::vector<T>>> query(Database db, std::string sql, Args&&... args);
// materializes rows, exactly like queryAsync — the future/task carries no lease

Task<Result<std::int64_t>>   executeBatch(Database db, std::string sql, /* mirrors the
                                          sync executeBatch/batchOf overloads */);

template <class Fn>  // T below = fn's Result<T> (or Task<Result<T>>) payload
Task<Result<T>>              transaction(Database db, Fn fn);
Task<Result<T>>              transaction(Database db, Isolation iso, Fn fn);

Task<Result<StreamingQuery>> queryStreaming(Database db, std::string sql, Args&&... args);

}  // namespace halcyon::coro
```

- **Error model:** every op returns `Task<Result<...>>`. Throwing style:
  `(co_await coro::query<T>(db, sql)).value()` — documented idiom; the
  overload set is not doubled.
- **Inherited behavior:** auto-retry of idempotent ops, reconnect, metrics,
  tracing, and logging all apply unchanged because each op is the existing
  public sync API executed against the bundle's sync view on the worker
  (§A.1/§A.2). OTel trace context is captured once, in `asyncBacking()` at
  the call point, and reinstated on the worker — the same helper and
  sequencing as the future-based path (parity verified in the plan).
- **Argument lifetime:** owned by the frame for everything except
  `LobSource` (see above); the forwarding references in the sketch exist
  only to move where possible — nothing reference-like survives into the
  lazy coroutine. Passing temporaries that die
  before the first `co_await` is safe by construction (adversarial
  unit test).
- Named and positional parameters are both supported, matching the sync
  overloads they forward to (LOB writes remain positional-only per v1.1).

### A.4 Async transaction scopes

`coro::transaction(db, fn)`:

1. Offload: acquire a pooled connection and `begin` (honoring the optional
   `Isolation` override) on a worker thread.
2. Run `fn(Transaction&)`:
   - **Plain callable returning `Result<T>`:** runs entirely on that worker.
     Inside it, use the normal blocking `tx.execute(...)`/`tx.query(...)` —
     the code is already on a worker thread; per-op awaitables inside a
     transaction would be pointless executor hops.
   - **`Task`-returning coroutine:** `co_await`ed, so callers can await
     non-Halcyon awaitables mid-transaction. A foreign awaitable controls
     where it resumes — after it, `fn` (and any `tx.*` call it then makes)
     may be running on a foreign thread, e.g. an event-loop thread. Two
     consequences: (a) the docs state plainly that every `tx.*` call is
     blocking on whatever thread `fn` currently occupies — awaiting foreign
     things mid-transaction moves that blocking cost onto the foreign
     runtime; (b) the scope itself **hops back to a Halcyon worker**
     (re-submitting its continuation through the §A.2 lock-post sequence)
     after `fn`'s task completes, so commit/rollback always execute on a
     worker regardless of where `fn` ended. The `Transaction` is only ever
     touched sequentially throughout, satisfying the single-owner contract.
     Edge case: if the hop-back submit fails because the executor is already
     gone (`InvalidState`), the scope rolls back synchronously on the
     current thread rather than leak an open transaction — documented.
   - Dispatch is by trait on `fn`'s invoke result.
3. Terminal semantics are exactly the sync `transaction(...)`'s: normal/`ok`
   return → commit; error return or exception → rollback, then propagate;
   restore/rollback failures poison so the pool discards the connection.

Holding a transaction (and its pooled connection) across long-lived awaits is
discouraged in the docs — the guide says to keep transactional scopes short.

### A.5 Async streaming

`StreamingQuery` wraps the v1.1 `StreamingResultSet` and owns it (and thereby
the connection lease). Move-only, single-owner, like the rest of the chain.

- `co_await sq.next()` → `std::optional<coro::StreamingRow>`, a thin wrapper
  over the v1.1 row. Row fetch offloads to the executor (it hits the server).
  `nullopt` means end **or** error; disambiguate with `sq.ok()`/`sq.error()`
  after the loop — the same mental model as the sync streaming API.
- **Scalar cells stay synchronous:** `row.get<T>(col)` forwards to the v1.1
  row and reads client-side data already fetched with the row; no executor
  hop.
- **LOB chunks are awaitable:** `row.lob(col)` returns
  `Result<coro::LobReader>` — the v1.1 sync method returns
  `Result<LobReader>` because invalid or out-of-order columns fail
  recoverably, and the async wrapper mirrors that. `coro::LobReader` exposes
  `Task<Result<std::size_t>> read(std::byte*, std::size_t)`
  (0 ⇒ EOF) and `isNull()` semantics identical to v1.1. Drain helpers
  `toVector()`, `toString()`, `toFile(path)` loop **once** on a worker
  (single offload for the whole drain) instead of one hop per chunk.
- Ascending-column-order enforcement, NULL-LOB semantics, and mid-stream
  abandonment behavior are all inherited from the v1.1 layer unchanged.
- **Borrowing rules for member awaitables.** `sq.next()`,
  `reader.read(buf, n)`, and the drain helpers return single-use awaitables
  that **borrow their receiver** (and `read` borrows the caller's buffer):
  the receiver must stay alive, unmoved, from the start of the await until
  it completes. The intended idiom is awaiting immediately
  (`co_await sq.next()`); storing a member awaitable and awaiting it later
  is documented as unsupported. A `coro::StreamingRow` borrows its
  `StreamingQuery` and is invalidated by the next `next()` — the same rule
  as the v1.1 sync row. These are the standard borrow constraints of the
  sync API carried over; they are stated explicitly in the header docs
  because lazy awaitables make it possible to violate them in ways the
  sync API cannot.

### A.6 Threading contract (documented prominently)

- Continuations resume on Halcyon executor worker threads. Don't run long
  CPU-bound work there; hop to your own thread/scheduler for that.
- Don't call `coro::syncWait` from an executor thread (self-deadlock risk;
  debug assert).
- `coro::transaction` commit/rollback always run on Halcyon workers (§A.4
  hop-back); user `tx.*` calls block whatever thread `fn` occupies at that
  moment.
- One connection/transaction/stream is used by at most one thread at a time
  across awaits — sequential use from different worker threads is within the
  existing single-owner contract (no concurrent access ever occurs). IBM
  permits a CLI handle to move between threads when access is coordinated
  and never concurrent; awaits sequence all access with happens-before
  edges through the executor. Verified under TSan in the stress suite.

## 4. Workstream B — Pool health stats

Additive, in the compiled C++17 library:

```cpp
struct PoolStats {
    // point-in-time
    std::size_t size;      // connections currently owned (idle + busy)
    std::size_t idle;
    std::size_t busy;
    std::size_t waiters;   // threads blocked in acquire()
    std::size_t peakBusy;  // high-water mark since construction
    // monotonic since pool construction
    std::uint64_t createdTotal;
    std::uint64_t acquiredTotal;
    std::uint64_t acquireTimeoutsTotal;
    std::uint64_t reapedIdleTotal;
    std::uint64_t reapedLifetimeTotal;
    std::uint64_t discardedTotal;  // poisoned + validation-failed discards
};

PoolStats ConnectionPool::stats() const;  // one snapshot under the pool mutex
PoolStats Database::poolStats() const;    // forwarder
```

- The snapshot is internally consistent (single lock acquisition): invariants
  `busy <= size` and `idle + busy == size` always hold within one snapshot.
- This is a programmatic accessor; the optional Prometheus adapter path is
  unchanged. Existing counters the pool already tracks for metrics are
  reused where they exist rather than duplicated.

## 6. Testing strategy

Per house rules: TDD throughout; **full suite including live-Db2 integration
before any completion claim** (Docker on Apple silicon needs
`DOCKER_DEFAULT_PLATFORM=linux/amd64`).

- **Unit — coroutine layer (MockCliDriver, C++20 target):** `Task` laziness,
  move semantics, exception propagation; symmetric transfer via a ~100k-deep
  await chain (no stack overflow); resume-on-worker asserted by thread id;
  result parity of every `coro::` op with its sync twin; transaction scopes
  with sync-fn and coroutine-fn on commit/rollback/exception/poison paths;
  async streaming chunk boundaries (empty LOB, NULL LOB, exact buffer
  multiple, mid-stream abandon); acquire-failure short-circuit; adversarial
  argument-lifetime test (temporaries destroyed before first await);
  `syncWait` debug assert on executor threads; weak-executor semantics
  (awaiting after the last `Database` copy is destroyed →
  `ErrorCode::InvalidState`, in-flight task drains safely); transaction
  hop-back (commit observed on a worker thread after `fn` resumes on a
  foreign thread, and the executor-gone rollback edge); exception
  containment (a throwing transaction callback / throwing user coroutine is
  rethrown from `co_await` after rollback — never escapes a worker thread).
- **Unit — C++17 support surface (MockCliDriver):** `asyncBacking()`
  trace-context capture-at-call-time and worker-side attachment parity with
  `executeAsync`; the pool-only sync view rejecting async entry points with
  `InvalidState`; sync-view destruction order (a dsn-opened `Database`
  destroyed while a task frame holds the bundle: the op completes, then
  teardown destroys pool before driver — run under ASan to prove no UAF on
  the pool's raw `ICliDriver*`); `Executor::post` ordering with `submit`;
  `Executor::onWorkerThread()` true on workers, false elsewhere;
  teardown-on-worker (the last `Database` copy destroyed inside a posted
  continuation: queue drains, the worker detaches instead of self-joining,
  TSan/ASan clean); the lock-vs-destroy race (a `State` lock taken on a
  worker while another thread destroys the last handle) neither deadlocks
  nor leaks.
- **Unit — parameter ownership:** a `std::string_view` of a buffer freed
  before the first await binds correctly (owned `Value` materialization);
  a **temporary** `LobSource` wrapper (`lobFile(path)` passed inline)
  streams correctly after the temporary dies; a `lobStream` source whose
  stream outlives the task round-trips; the borrow contract is exercised
  by the integration LOB tests.
- **Unit — pool stats (C++17 target):** counter transitions across
  acquire/release/timeout/reap/discard; snapshot invariants; `peakBusy`
  high-water behavior.
- **Integration (live Db2):** coro round-trips for query/execute/batch;
  async transaction commit/rollback visibility from a second connection;
  async streaming LOB ≥ 32 MiB checksum round-trip in both directions;
  N concurrent coroutine chains through the pool.
- **Stress (opt-in suite, TSan + ASan):** a scenario driving concurrent
  coroutine chains (queries, transactions, streaming) mixed with `stats()`
  polling — data-race freedom, pool-state restoration, stats invariants
  under load.

## 7. Risks & subtleties

- **Compiler coroutine maturity.** Older toolchains have real coroutine
  bugs. Documented floor: GCC 11+, Clang 14+, AppleClang 14+; the C++20
  targets are feature-gated and skip cleanly below the floor.
- **ODR across mixed-standard TUs.** The defense is that existing class
  definitions are never *conditionally* different: the C++17 support surface
  (`asyncBacking`, `post`, `onWorkerThread`, `PoolStats`) consists of
  unconditional members identical in every TU, while everything
  C++20-gated lives exclusively in `halcyon/coro.hpp` as free
  functions/types. `#if`-gated members on `Database` — which would give
  C++17 and C++20 TUs different class definitions — remain forbidden.
- **Lazy tasks + dangling references** — the classic coroutine bug.
  Countered structurally (non-coroutine factories materializing owned seam
  `Value`s — a decay-copy alone would leave `string_view` non-owning) and
  by the adversarial lifetime test. `LobSource` is the documented borrowed
  exception; member awaitables carry explicit borrow rules (§A.5).
- **`syncWait` self-deadlock** on executor threads when the pool is
  saturated — documented plus debug assert.
- **Symmetric transfer** is required, not optional: naive resumption
  overflows the stack on long chains; the deep-chain unit test is the gate.
- **OTel context propagation parity** between futures and tasks — the
  factory captures with the same helper and sequencing as `executeAsync`,
  verified by test, so traces don't silently lose spans when users migrate
  to `co_await`.
- **Executor lifetime and self-join.** User continuations run on worker
  threads, which creates three related failure modes: a strong executor
  reference in a frame or queued closure (ownership cycle / destructor on
  worker), a *temporary* strong reference in `await_suspend` losing a race
  and becoming the final release on a worker, and a user's own `Database`
  copy dying in a frame on a worker. The first two are removed structurally
  by referencing only `Executor::State` (releasing it controls no thread
  lifetimes); the third cannot be prevented by the coro layer at all, which
  is why teardown itself is made self-safe (handle/`State` split,
  join-others/detach-self). Await-after-death is a defined `InvalidState`
  error; the trade-off — a lazy task can outlive its backing only until
  awaited — is documented.
- **Exception containment on workers.** The worker loop invokes jobs bare;
  only `submit`'s `packaged_task` contains exceptions today. `post` is
  therefore contracted noexcept-callable, and every coro closure earns that
  by catching into the frame's `std::exception_ptr` and rethrowing from
  `co_await` — a throwing user callback must surface at the await, never as
  `std::terminate` on a worker. Unit-tested directly.
- **Foreign resumption in coroutine-fn transactions.** A non-Halcyon
  awaitable resumes wherever its runtime chooses; without countermeasures,
  commit/rollback would block that thread. The §A.4 hop-back confines
  terminal operations to Halcyon workers; mid-`fn` `tx.*` calls remain the
  caller's responsibility and are documented as blocking.
- **Blocking CLI underneath:** the coroutine layer changes ergonomics, not
  concurrency limits — throughput is still bounded by executor threads ×
  blocking CLI calls. The docs say so explicitly to set expectations.
