# Halcyon v1.2 — Coroutines & Async Streaming — Design Spec

**Date:** 2026-07-16 · **Status:** approved (design) · **Baseline:** v1.1.0
(2026-07-13)

## 1. Overview

v1.2 delivers the marquee item deferred since the original design: the C++20
coroutine layer the executor seam was built "coroutine-ready" for, including
the async streaming follow-up (`queryStreamingAsync`) recorded in the v1.1
spec §10. One operational ride-along completes the backlog's last small item.

1. **C++20 coroutine layer** — `halcyon::coro::Task<T>` plus awaitable forms
   of `execute`, `query`, `executeBatch`, `transaction`, and streaming
   (awaitable row fetch and LOB chunk reads).
2. **Pool health stats** — a `PoolStats` snapshot accessor on
   `ConnectionPool` and `Database`.

**Approach (user-approved):** executor-offload awaitables. Each awaited
operation submits the existing blocking call to the existing `Executor`
thread pool; the awaiting coroutine suspends and is resumed on the completing
worker thread. The whole layer is a **header-only C++20 adapter over the
public API** — the compiled library stays C++17. Rejected alternatives: a
per-connection async actor/strand subsystem (large new machinery with no real
concurrency gain over a blocking CLI) and caller-supplied resume schedulers /
sender-receiver shapes (bigger abstract surface; can be layered on later
without breaking changes).

### Goals

- Full async parity: everything the `std::future` API covers, plus streaming,
  awaitable from C++20 coroutines.
- Library contracts intact: dual error model, RAII/single-owner semantics,
  thread-safe pool, auto-retry/metrics/tracing/logging inherited unchanged
  (each awaitable *is* the existing sync path, offloaded).
- The compiled library, its ABI, and its C++17 requirement are unchanged;
  the coroutine layer is a separate opt-in header for C++20 consumers.
- Zero new dependencies. SemVer **minor**: purely additive, version `1.2.0`.

### Non-goals (explicitly out of scope)

- Cancellation tokens / stop sources for in-flight operations.
- Caller-supplied resume schedulers; asio/libcoro/stdexec interop.
- Raising the library's own build standard to C++20.
- Generator-style `co_for` row iteration (use the `next()` loop).
- LOB support in `executeBatch`; scrollable cursors; vcpkg/Conan submission;
  Windows/AIX; versioned docs (all remain on the recorded backlog).

## 2. Decisions (locked)

| Topic | Decision |
|---|---|
| Version | `1.2.0`, SemVer minor, purely additive |
| Task type | Own dependency-free `halcyon::coro::Task<T>` (lazy, move-only, single-consumer, symmetric transfer); no third-party runtime |
| Execution model | Offload to the existing `Executor`; resume on the completing worker thread |
| Packaging | New header `halcyon/coro.hpp`, **not** in the `halcyon.hpp` umbrella; clear `#error` without C++20 coroutine support |
| ODR safety | No `#if`-gated members on existing classes; all coroutine API is free functions/types in the new header, namespace `halcyon::coro` |
| Error model | All ops return `Task<Result<...>>`; throwing style is the documented idiom `(co_await …).value()` — no doubled overload set |
| Argument lifetime | Tasks are lazy ⇒ all arguments (SQL, params) copied into the coroutine frame at call time |
| Transaction fn | `coro::transaction` accepts a plain `Result<T>`-returning callable **or** a `Task`-returning coroutine (trait-dispatched) |
| Async streaming | `StreamingQuery` wraps the v1.1 `StreamingResultSet`; awaitable `next()` and LOB `read()`; scalar cell reads stay synchronous |
| Sync bridge | `coro::syncWait(task)`; forbidden (documented + debug assert) on executor worker threads |
| Pool stats | `PoolStats` value struct; `ConnectionPool::stats()` snapshot under the pool mutex; `Database::poolStats()` forwarder |
| Seam growth | **None.** Zero new `ICliDriver` methods; the layer sits entirely above the facade |
| Compiler floor (coro tests/examples) | GCC 11+, Clang 14+, AppleClang 14+; gated on the coroutine feature-test macros, skipped cleanly below |

## 3. Workstream A — Coroutine layer

### A.1 Packaging & build

- New public header `include/halcyon/coro.hpp` (implementation may split into
  `include/halcyon/coro/*.hpp` detail headers; `coro.hpp` is the entry point).
- Header-only: templates/inline over `Database`, `ConnectionPool`,
  `Executor::submit`, and `Result`. No new symbols in the compiled library,
  no ABI change, no new consumer-facing CMake options.
- Guard at the top of the header: if `__cpp_impl_coroutine` /
  `__cpp_lib_coroutine` are absent, `#error "halcyon/coro.hpp requires C++20
  coroutines; compile this TU with -std=c++20 or newer"`.
- The umbrella `halcyon.hpp` does **not** include it; C++17 consumers see
  nothing new.
- Test/example targets that exercise the layer compile with `cxx_std_20` and
  are excluded with a configure-time notice when the toolchain lacks
  coroutine support.

### A.2 `Task<T>` machinery

- `halcyon::coro::Task<T>`: lazy (nothing runs until first `co_await`),
  move-only, single-consumer; `Task<void>` specialization.
- Completion resumes the awaiter via **symmetric transfer** — deep task
  chains must not grow the stack (unit-tested with a ~100k-deep chain).
- Exceptions thrown inside a task propagate out of `co_await` (rethrown), so
  the throwing style composes naturally.
- A `Task` that is never awaited never runs and destroys cleanly. Once
  awaited, the consumer is suspended until completion, so mid-flight
  destruction by the consumer cannot occur; no detach/fire-and-forget API.
- `coro::syncWait(Task<T>)` bridges to synchronous code (blocks on a
  condition variable, returns `T` or rethrows). Calling it from an executor
  worker thread can self-deadlock when the pool is saturated: documented,
  and debug builds assert (the `Executor` can identify its own threads).

### A.3 Awaitable operations

All entry points mirror the `std::future` async API and capture the shared
pool/executor (`shared_ptr` copies), so a `Task` may outlive the `Database`
object — the same guarantee today's futures give.

```cpp
namespace halcyon::coro {

Task<Result<std::int64_t>>   execute(Database db, std::string sql, Args&&... args);

template <class T>
Task<Result<std::vector<T>>> query(Database db, std::string sql, Args&&... args);
// materializes rows, exactly like queryAsync — the future/task carries no lease

Task<Result<std::int64_t>>   executeBatch(Database db, std::string sql, /* mirrors the
                                          sync executeBatch/batchOf overloads */);

template <class Fn>  // T below = fn's Result<T> (or Task<Result<T>>) payload
Task<Result<T>>              transaction(Database db, Fn fn);
Task<Result<T>>              transaction(Database db, Isolation iso, Fn fn);

Task<Result<StreamingQuery>> queryStreaming(Database db, std::string sql, Args&&... args);

}  // namespace halcyon::coro
```

- **Error model:** every op returns `Task<Result<...>>`. Throwing style:
  `(co_await coro::query<T>(db, sql)).value()` — documented idiom; the
  overload set is not doubled.
- **Inherited behavior:** auto-retry of idempotent ops, reconnect, metrics,
  tracing, and logging all apply unchanged because the offloaded body is the
  existing sync path. OTel trace context is captured at the call point and
  reinstated on the worker, reusing the same context-capture helper the
  future-based async path uses (parity verified in the plan).
- **Argument lifetime:** tasks are lazy, so `sql` and every parameter are
  decay-copied into the coroutine frame at call time (the forwarding
  references above exist only to move where possible). Passing temporaries that die
  before the first `co_await` is safe by construction (adversarial
  unit test).
- Named and positional parameters are both supported, matching the sync
  overloads they forward to (LOB writes remain positional-only per v1.1).

### A.4 Async transaction scopes

`coro::transaction(db, fn)`:

1. Offload: acquire a pooled connection and `begin` (honoring the optional
   `Isolation` override) on a worker thread.
2. Run `fn(Transaction&)`:
   - **Plain callable returning `Result<T>`:** runs entirely on that worker.
     Inside it, use the normal blocking `tx.execute(...)`/`tx.query(...)` —
     the code is already on a worker thread; per-op awaitables inside a
     transaction would be pointless executor hops.
   - **`Task`-returning coroutine:** `co_await`ed, so callers can await
     non-Halcyon awaitables mid-transaction. After each resumption the
     coroutine may be on a different worker thread; the `Transaction` is only
     ever touched sequentially, which satisfies the single-owner contract.
   - Dispatch is by trait on `fn`'s invoke result.
3. Terminal semantics are exactly the sync `transaction(...)`'s: normal/`ok`
   return → commit; error return or exception → rollback, then propagate;
   restore/rollback failures poison so the pool discards the connection.

Holding a transaction (and its pooled connection) across long-lived awaits is
discouraged in the docs — the guide says to keep transactional scopes short.

### A.5 Async streaming

`StreamingQuery` wraps the v1.1 `StreamingResultSet` and owns it (and thereby
the connection lease). Move-only, single-owner, like the rest of the chain.

- `co_await sq.next()` → `std::optional<coro::StreamingRow>`, a thin wrapper
  over the v1.1 row. Row fetch offloads to the executor (it hits the server).
  `nullopt` means end **or** error; disambiguate with `sq.ok()`/`sq.error()`
  after the loop — the same mental model as the sync streaming API.
- **Scalar cells stay synchronous:** `row.get<T>(col)` forwards to the v1.1
  row and reads client-side data already fetched with the row; no executor
  hop.
- **LOB chunks are awaitable:** `row.lob(col)` returns `coro::LobReader`, an
  async wrapper with `Task<Result<std::size_t>> read(std::byte*, std::size_t)`
  (0 ⇒ EOF) and `isNull()` semantics identical to v1.1. Drain helpers
  `toVector()`, `toString()`, `toFile(path)` loop **once** on a worker
  (single offload for the whole drain) instead of one hop per chunk.
- Ascending-column-order enforcement, NULL-LOB semantics, and mid-stream
  abandonment behavior are all inherited from the v1.1 layer unchanged.

### A.6 Threading contract (documented prominently)

- Continuations resume on Halcyon executor worker threads. Don't run long
  CPU-bound work there; hop to your own thread/scheduler for that.
- Don't call `coro::syncWait` from an executor thread (self-deadlock risk;
  debug assert).
- One connection/transaction/stream is used by at most one thread at a time
  across awaits — sequential use from different worker threads is within the
  existing single-owner contract (no concurrent access ever occurs). Verified
  under TSan in the stress suite.

## 4. Workstream B — Pool health stats

Additive, in the compiled C++17 library:

```cpp
struct PoolStats {
    // point-in-time
    std::size_t size;      // connections currently owned (idle + busy)
    std::size_t idle;
    std::size_t busy;
    std::size_t waiters;   // threads blocked in acquire()
    std::size_t peakBusy;  // high-water mark since construction
    // monotonic since pool construction
    std::uint64_t createdTotal;
    std::uint64_t acquiredTotal;
    std::uint64_t acquireTimeoutsTotal;
    std::uint64_t reapedIdleTotal;
    std::uint64_t reapedLifetimeTotal;
    std::uint64_t discardedTotal;  // poisoned + validation-failed discards
};

PoolStats ConnectionPool::stats() const;  // one snapshot under the pool mutex
PoolStats Database::poolStats() const;    // forwarder
```

- The snapshot is internally consistent (single lock acquisition): invariants
  `busy <= size` and `idle + busy == size` always hold within one snapshot.
- This is a programmatic accessor; the optional Prometheus adapter path is
  unchanged. Existing counters the pool already tracks for metrics are
  reused where they exist rather than duplicated.

## 6. Testing strategy

Per house rules: TDD throughout; **full suite including live-Db2 integration
before any completion claim** (Docker on Apple silicon needs
`DOCKER_DEFAULT_PLATFORM=linux/amd64`).

- **Unit — coroutine layer (MockCliDriver, C++20 target):** `Task` laziness,
  move semantics, exception propagation; symmetric transfer via a ~100k-deep
  await chain (no stack overflow); resume-on-worker asserted by thread id;
  result parity of every `coro::` op with its sync twin; transaction scopes
  with sync-fn and coroutine-fn on commit/rollback/exception/poison paths;
  async streaming chunk boundaries (empty LOB, NULL LOB, exact buffer
  multiple, mid-stream abandon); acquire-failure short-circuit; adversarial
  argument-lifetime test (temporaries destroyed before first await);
  `syncWait` debug assert on executor threads.
- **Unit — pool stats (C++17 target):** counter transitions across
  acquire/release/timeout/reap/discard; snapshot invariants; `peakBusy`
  high-water behavior.
- **Integration (live Db2):** coro round-trips for query/execute/batch;
  async transaction commit/rollback visibility from a second connection;
  async streaming LOB ≥ 32 MiB checksum round-trip in both directions;
  N concurrent coroutine chains through the pool.
- **Stress (opt-in suite, TSan + ASan):** a scenario driving concurrent
  coroutine chains (queries, transactions, streaming) mixed with `stats()`
  polling — data-race freedom, pool-state restoration, stats invariants
  under load.

## 7. Risks & subtleties

- **Compiler coroutine maturity.** Older toolchains have real coroutine
  bugs. Documented floor: GCC 11+, Clang 14+, AppleClang 14+; the C++20
  targets are feature-gated and skip cleanly below the floor.
- **ODR across mixed-standard TUs.** The reason for the free-function/
  separate-header design: `#if`-gated members on `Database` would give C++17
  and C++20 TUs different class definitions. The layer never modifies
  existing headers' class definitions.
- **Lazy tasks + dangling references** — the classic coroutine bug. Countered
  structurally (by-value frames) and by the adversarial lifetime test.
- **`syncWait` self-deadlock** on executor threads when the pool is
  saturated — documented plus debug assert.
- **Symmetric transfer** is required, not optional: naive resumption
  overflows the stack on long chains; the deep-chain unit test is the gate.
- **OTel context propagation parity** between futures and tasks — same
  capture helper, verified by test, so traces don't silently lose spans when
  users migrate to `co_await`.
- **Executor lifetime:** tasks hold `shared_ptr<Executor>` (as futures do
  today), so a task outliving its `Database` remains safe; executor
  destruction still drains in-flight work.
- **Blocking CLI underneath:** the coroutine layer changes ergonomics, not
  concurrency limits — throughput is still bounded by executor threads ×
  blocking CLI calls. The docs say so explicitly to set expectations.
