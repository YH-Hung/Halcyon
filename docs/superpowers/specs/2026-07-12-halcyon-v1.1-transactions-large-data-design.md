# Halcyon v1.1 — Transactions & Large Data — Design Spec

**Date:** 2026-07-12 · **Status:** approved (brainstormed + user-approved) ·
**Baseline:** v1.0.0 (2026-07-01)

## 1. Overview

v1.1 finishes the transaction story and removes the "LOB values must fit in
memory" ceiling, plus one operational ride-along. Four features, all drawn from
the deferred backlog recorded in the v1 specs:

1. **Savepoints & nested transactions** — `SAVEPOINT` / `ROLLBACK TO SAVEPOINT`
   surfaced as a RAII `Savepoint` guard and an explicit `Transaction::nested(...)`
   scope.
2. **Isolation-level API** — Db2 isolation levels (UR/CS/RS/RR) as a connection
   default and a per-transaction override.
3. **LOB streaming** — chunked read (`SQLGetData`) and write
   (data-at-exec/`SQLPutData`) of BLOB/CLOB values with O(chunk) memory.
4. **Pluggable structured logging** — an `ILogger` interface in the
   observability layer, following the exact metrics/tracing pattern.

**Approach (user-approved):** native CLI mapping with minimal seam growth.
Savepoints are plain SQL through the existing seam; isolation adds one seam
method; LOB streaming adds three coarse-grained seam methods with the CLI
`SQL_NEED_DATA` state machine hidden inside `Db2CliDriver`. Rejected
alternatives: SQL-only emulation (no true streaming; per-chunk server
round-trips) and a general streaming-abstraction layer (YAGNI; oversized
public surface).

### Goals

- Ship the four features above with the library's existing contracts intact:
  dual error model, RAII/single-owner semantics, thread-safe pool, C++17 only.
- Keep the "thin CLI seam" invariant: no `sqlcli1.h` types above
  `detail::cli`; every new seam method mockable in unit tests.
- Zero new hard dependencies; logging follows the optional-observability
  pattern (interface + null default, zero overhead when unset).
- SemVer **minor** release: purely additive public API; version `1.1.0`.

### Non-goals (explicitly out of scope)

- Scrollable/positionable cursors (forward-only streaming remains the model).
- C++20 coroutine layer; async variants of the new streaming API
  (`queryStreaming` is sync-only in v1.1).
- Package-registry submission (vcpkg/Conan), Windows/AIX, versioned docs.
- Standalone pool health-stats accessor (not selected for this release).
- LOB support inside `executeBatch` (array binding + data-at-exec is rejected
  with a clear error).
- XML / DECFLOAT or other new column-type work beyond BLOB/CLOB streaming.

## 2. Decisions (locked)

| Topic | Decision |
|---|---|
| Version | `1.1.0`, SemVer minor; `v1.0.0` tag must be cut first as baseline |
| Savepoints | Plain SQL via existing seam; RAII `Savepoint`; explicit `Transaction::nested(fn)` (no auto-detection) |
| Savepoint names | Auto `halcyon_sp_<n>`; explicit names validated as SQL identifiers (injection defense) |
| Isolation enum | Db2 terminology: `UncommittedRead, CursorStability, ReadStability, RepeatableRead` |
| Isolation knobs | `ConnectionParams::isolation` default + per-transaction override, restored on transaction end |
| LOB read | `queryStreaming` → row-at-a-time cursor; pull-based `LobReader` over `SQLGetData`; ascending-column-order enforced |
| LOB write | `LobSource` parameter wrappers (`lobFile`/`lobStream`/`lobCallback`, `.asClob()` for CLOB targets); positional binds only; driver-internal `SQLParamData`/`SQLPutData` loop |
| Seam growth | `setIsolation`, `fetchNext`, `getValue`, `getDataChunk`, `executeStreaming` — five new `ICliDriver` methods total |
| Logging | `ILogger` + `LogField` in `observability/`; wired via `ObservabilityConfig`; `NullLogger` default + `StderrLogger` reference adapter |
| `ICliDriver` compat | New pure virtuals accepted; `detail/` is excluded from the public compatibility policy (changelog notes it) |

## 3. Workstream A — Structured logging seam

Built first so the other workstreams can add log points as they land.

### A.1 Interface

New header `include/halcyon/observability/logging.hpp`:

```cpp
enum class LogLevel { Debug, Info, Warn, Error };

struct LogField {
    std::string_view key;
    std::variant<std::int64_t, double, bool, std::string_view> value;
};

class ILogger {
public:
    virtual ~ILogger() = default;
    // Must not throw; called on hot paths only behind a null check.
    virtual void log(LogLevel level, std::string_view event,
                     std::initializer_list<LogField> fields) noexcept = 0;
};
```

C++17 constraint: `std::initializer_list<LogField>` (not `std::span`). Fields
reference stack temporaries — adapters must copy inside `log()`, documented on
the interface.

### A.2 Wiring & adapters

- `ObservabilityConfig` gains `std::shared_ptr<ILogger> logger;` (null =
  disabled), flowing through `PoolConfig` exactly like metrics/tracing today.
- Internal call sites use a small inline helper that is a single branch when
  no logger is set.
- `StderrLogger` reference adapter (no dependencies): logfmt-style
  `ts=… level=… event=… key=value…`, one buffered `fwrite` per line so
  concurrent lines don't interleave. spdlog/syslog/etc. are user-side
  `ILogger` implementations, not Halcyon dependencies.

### A.3 Log points (initial set)

| Event | Level |
|---|---|
| `connect.ok` / `connect.fail` | Info / Error |
| `reconnect.attempt` / `reconnect.ok` / `reconnect.fail` | Warn / Info / Error |
| `retry.attempt` (op, attempt, backoff_ms) | Warn |
| `pool.exhausted` (waiters, timeout_ms) | Warn |
| `pool.reap` (idle/lifetime, count) | Debug |
| `stmt_cache.evict` (sql_hash) | Debug |
| `txn.poisoned` / `savepoint.poisoned` | Error |
| `isolation.restore_fail` | Error |

## 4. Workstream B — Isolation-level API

### B.1 Public API

```cpp
enum class Isolation { UncommittedRead, CursorStability, ReadStability, RepeatableRead };

// Connection/pool default (applied at connect):
params.isolation = Isolation::CursorStability;   // std::optional; nullopt = server default

// Per-transaction override (all existing entry points gain an overload):
auto tx  = conn.begin(Isolation::RepeatableRead);
db.transaction(Isolation::ReadStability, [&](Transaction& tx) { ... });
halcyon::transaction(db, Isolation::ReadStability, fn);
```

The enum uses **Db2 terminology** deliberately. The CLI mapping (inside
`detail::cli` only) is the classic gotcha this avoids exposing:
UR→`SQL_TXN_READ_UNCOMMITTED`, CS→`SQL_TXN_READ_COMMITTED`,
RS→`SQL_TXN_REPEATABLE_READ`, RR→`SQL_TXN_SERIALIZABLE`.

### B.2 Semantics

- One new seam method: `Result<void> setIsolation(ConnectionHandle, Isolation)`
  → `SQLSetConnectAttr(SQL_ATTR_TXN_ISOLATION, …)`.
- `ConnectionParams::isolation` is applied once at connect; every pooled
  connection carries the default.
- A per-transaction override sets the attribute immediately before the
  transaction begins (no open transaction at that point, per CLI rules) and
  restores the prior level when the transaction ends by **any** path (commit,
  rollback, destructor).
- If the restore fails, the transaction is **poisoned** (same mechanism as the
  existing autocommit-restore failure), so a connection can never return to
  the pool with the wrong isolation level.

## 5. Workstream C — Savepoints & nested transactions

### C.1 Public API

```cpp
Result<Savepoint> sp = tx.savepoint();            // auto name: halcyon_sp_<n>
Result<Savepoint> sp = tx.savepoint("stage1");    // validated explicit name

sp->rollback();   // ROLLBACK TO SAVEPOINT <name>  (one-shot)
sp->release();    // RELEASE SAVEPOINT <name>      (one-shot)
// Destructor with neither called: rollback-to + release (undo by default).

tx.nested([&](Transaction& inner) -> Result<void> { ... });
```

`Savepoint` mirrors `Transaction`'s contract: movable, non-copyable,
single-owner, one-shot, `poisoned()` on failed terminal operations. The
destructor is `noexcept`; a destructor-path failure poisons the owning
transaction (conservative: the pool discards the connection rather than risk
inconsistent state).

`nested(fn)` wraps `fn` in a savepoint scope: normal/`ok()` return →
`RELEASE`; error return or exception → `ROLLBACK TO SAVEPOINT`, then the
error/exception propagates. `fn` receives the **same** `Transaction&`;
calling `commit()`/`rollback()` on it inside a nested scope is a programming
error — `nested` detects a no-longer-active transaction after `fn` and
returns `ErrorCode::InvalidState`. Nesting is explicit; `transaction(db, fn)`
does **not** silently convert to a savepoint.

### C.2 Implementation

- **Zero new seam methods.** The three statements are executed as plain SQL
  through the existing prepare/execute path:
  `SAVEPOINT <name> ON ROLLBACK RETAIN CURSORS` (the clause is mandatory in
  Db2 LUW), `ROLLBACK TO SAVEPOINT <name>`, `RELEASE SAVEPOINT <name>`.
- Savepoint statements cannot be parameterized, so the name is interpolated —
  therefore names are strictly validated: `[A-Za-z_][A-Za-z0-9_]{0,127}`,
  case-insensitively not starting with `SYS` (Db2 reserves it). Anything else
  → `ErrorCode::InvalidArgument`. This is the injection defense; it is a hard
  gate, not a convention.
- Auto names come from a per-transaction counter (`halcyon_sp_1`, …), so they
  never collide within a transaction.
- Db2 behavior verified early by integration tests (see §8): nested savepoint
  support, name-reuse rules, and `ROLLBACK TO` destroying later savepoints.
  Destruction order (inner before outer) is documented; destroying out of
  order after an outer `rollback()` poisons conservatively.

## 6. Workstream D — LOB streaming

The largest workstream. Memory guarantee for both directions: **O(chunk)**,
never O(LOB).

### D.1 Read path

```cpp
auto rs = conn.queryStreaming("SELECT id, content FROM docs WHERE id = ?", id);
while (auto row = rs->next()) {
    auto id = row->get<std::int64_t>(0);          // scalar cells still typed
    LobReader lob = row->lob(1);
    std::array<std::byte, 256 * 1024> buf;
    for (;;) {
        auto n = lob.read(buf.data(), buf.size());   // Result<std::size_t>
        if (!n.ok()) return n.error();
        if (*n == 0) break;                          // EOF
        sink.write(..., *n);
    }
}
```

- `queryStreaming` exists wherever `query` exists; the returned
  `StreamingResultSet` owns the connection lease for its lifetime, like
  today's `ResultSet`.
- It uses a dedicated row-at-a-time path (an extension of the existing
  long-column fallback from the rowset-fetch design) — **no block binding**,
  so `SQLGetData` is legal on every column.
- CLI requires ascending column access in this mode. `StreamingRow` enforces
  it: reading a column at or before the last-accessed index returns
  `ErrorCode::InvalidState` with a message naming the constraint.
- `StreamingResultSet::next()` returns `std::optional<StreamingRow>`
  (`nullopt` = end **or** error); like today's `ResultSet`, check
  `ok()`/`error()` after the loop to distinguish the two. Scalar cells go
  through a typed `getValue` seam method so the existing `TypeBinder`/`Value`
  mapping applies unchanged.
- `LobReader`: `Result<std::size_t> read(std::byte*, std::size_t)` (0 ⇒ EOF),
  `bool isNull()`, plus convenience sinks `toString()`, `toVector()`,
  `toStream(std::ostream&)`, `toFile(const std::string&)`. Byte-oriented;
  CLOBs stream as UTF-8 text chunks (`SQL_C_CHAR`, terminator handled inside
  the driver). SQL NULL is only discoverable from the first driver chunk, so
  `isNull()` is meaningful after the first `read()` — a NULL LOB reads as
  immediate EOF (0) with `isNull()` then true.
- Abandoning a `LobReader` or `StreamingResultSet` mid-stream is safe: cursor
  close discards pending data.

### D.2 Write path

```cpp
conn.execute("INSERT INTO docs(id, content) VALUES (?, ?)",
             id, halcyon::lobFile("report.pdf"));
// also: halcyon::lobStream(istream&, sizeHintBytes)   — sizeHint optional
//       halcyon::lobCallback(fn, sizeHintBytes)       — fn(std::byte*, cap) -> size_t, 0 = EOF
```

- `LobSource` is a parameter wrapper accepted alongside ordinary positional
  binds. Binding any `LobSource` routes the statement through the
  `executeStreaming` seam method. **Positional only:** the named-`params`
  class erases values to the seam `Value` variant, which cannot carry a
  stream — mixing `LobSource` with named parameters is unsupported
  (documented; use positional binds for LOB statements).
- The driver cannot infer the target column type, so sources default to
  binary (`SQL_BLOB`); chain `.asClob()` to bind as `SQL_CLOB`/`SQL_C_CHAR`
  for character LOB columns.
- Data-at-exec works without a total size (`SQL_DATA_AT_EXEC`); the size hint,
  when present, is passed through for driver optimization.
- The driver pulls from the source into a fixed internal staging buffer
  (256 KiB) and `SQLPutData`s per chunk. A source failure aborts the execute
  with a mapped error and the statement is cancelled (`SQLCancel`) so a cached
  statement handle is never left in `SQL_NEED_DATA` state.
- `executeBatch` + `LobSource` is rejected by the existing batch validation
  with a clear error (documented non-goal).

### D.3 Seam changes (all of them)

```cpp
struct GetDataChunk { std::size_t bytes; bool done; bool isNull; };

virtual Result<void>          setIsolation(ConnectionHandle, Isolation) = 0;      // §4
virtual Result<bool>          fetchNext(StatementHandle) = 0;                     // row available?
virtual Result<Value>         getValue(StatementHandle, std::size_t col) = 0;     // scalar cell, typed
virtual Result<GetDataChunk>  getDataChunk(StatementHandle, std::size_t col,
                                           std::byte* buf, std::size_t bufLen) = 0;
virtual Result<std::int64_t>  executeStreaming(StatementHandle,
                                               const std::vector<Value>& params,
                                               std::vector<ParamStreamSource> sources) = 0;
```

`executeStreaming` receives the full positional `params` vector (streamed
positions hold `Null` placeholders that the driver ignores) plus the stream
`sources`, and binds both regular and data-at-exec parameters in one pass.
`ParamStreamSource` is a seam-level value type pairing the 0-based parameter
index with a pull callback (`std::function<std::size_t(std::byte*,
std::size_t)>`, returning 0 at EOF and `ParamStreamSource::npos` on source
failure), an optional byte-size hint, and an `isClob` flag; the public
`LobSource` wrappers lower to it at bind time.

The `SQL_NEED_DATA`/`SQLParamData`/`SQLPutData` state machine lives entirely
inside `Db2CliDriver::executeStreaming`; above the seam it is one call.
`MockCliDriver` implements all four with scriptable chunk boundaries.

## 8. Testing strategy

Per house rules: TDD throughout; **full suite including live-Db2 integration
before any completion claim** (Docker on Apple silicon needs
`DOCKER_DEFAULT_PLATFORM=linux/amd64`).

- **Unit (MockCliDriver):** every new seam method; savepoint SQL emission and
  name validation (including `SYS` prefix and injection attempts); RAII
  semantics for `Savepoint` on all exit paths; isolation set/restore on
  commit/rollback/destructor and poison-on-restore-failure; `LobReader` chunk
  boundaries (empty LOB, NULL LOB, exact buffer multiple, mid-stream abandon);
  `executeStreaming` NEED_DATA sequencing and source-failure cancellation;
  ascending-column-order enforcement; logging via a `CapturingLogger` test
  double (events fire, null logger costs one branch).
- **Integration (live Db2, first task of the savepoint workstream):** verify
  Db2's actual savepoint semantics — nested savepoints, name reuse,
  `ROLLBACK TO` destroying later savepoints — then behavioral tests: nested
  scope rollback restores pre-savepoint data while outer commit persists the
  rest; UR sees an uncommitted change from a second connection while CS does
  not; LOB round-trips ≥ 32 MiB in both directions verified by checksum, via
  file, stream, and callback sources.
- **Stress (opt-in suite, TSan + ASan):** a scenario mixing concurrent
  transactions with savepoints and per-transaction isolation overrides,
  verifying pool state (isolation/autocommit) is always restored or the
  connection discarded.

## 9. Risks & subtleties

- **Db2 savepoint semantics** vary by version (nested support, name rules).
  The integration spike runs *first* in that workstream so the API hardens
  against reality, not documentation.
- **Savepoint name interpolation** is the one place v1.1 builds SQL from user
  input; the identifier whitelist is a security gate and gets adversarial
  unit tests.
- **`SQLGetData` ordering** and its incompatibility with block-fetch binding
  is why streaming is a separate statement path (rowset size 1, no column
  binds); the existing long-column fallback already proves this path.
- **Statement cache × data-at-exec:** an aborted streaming execute must
  `SQLCancel` so a cached handle never rejoins the cache in `SQL_NEED_DATA`
  state; a unit test scripts exactly this sequence.
- **Isolation restore failure** must poison (never return a wrong-isolation
  connection to the pool) — same rationale as the existing autocommit poison.
- **`initializer_list<LogField>` lifetime:** fields are valid only during
  `log()`; the interface docs require adapters to copy. `StderrLogger`
  demonstrates the pattern.
- **`ICliDriver` gains pure virtuals** — breaking for out-of-tree seam
  implementations; accepted because `detail/` is outside the compatibility
  policy, and called out in the changelog.
